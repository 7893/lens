# 关于 Evolution 重构逻辑的深度解答 (God-Level FAQ)

针对你提出的两个关键架构问题，以下是目前“旗舰版”代码实现的底层设计逻辑：

---

## 1. 任务分支逻辑：为什么选择“新增分支”而非“逻辑复用”？

在 \`workflow.ts\` 中，我们采用了 **「逻辑分支解耦，原子服务共享」** 的策略。

- **逻辑分支 (\`if (type === 'refresh-photo')\`)**:
  - **理由**: \`process-photo\`（新图入库）涉及昂贵的原图下载、多画质 R2 存储及 14 个数据库字段的初始化。而 \`refresh-photo\`（老图刷新）本质上是**原地升级**。
  - **极致优化**: 刷新任务被禁止触发任何外部网络下载请求。它直接从本地 R2 取流。这不仅节省了 Unsplash API 配额，还将任务延迟降低了 60% 以上。
- **服务共享**:
  - 虽然分支不同，但它们在分析层完全复用了 \`services/ai.ts\` 里的 \`analyzeImage\` (Llama 4) 和 \`generateEmbedding\` (BGE-M3)。
  - **结论**: 这种结构在保证了功能差异化的同时，维持了 AI 理解逻辑的绝对一致性。

## 2. 财务审计调度：如何对抗 GraphQL 调用频率问题？

我们遵循 **「按需审计，拒绝空转」** 的经济学原则。

- **执行窗口**: 财务审计与任务派发逻辑被严格锁定在 **UTC 23:00** 这一特定时段。
- **频率保护**:
  - 系统在白天的 23 个小时内处于“财务休眠”状态，完全不调用 GraphQL API，将资源预留给搜索和采集。
  - **极致收窄**: 目前代码建议锁定在 \`currentMinute < 15\` 或更精准的 \`minute === 0\`。这意味着每天全系统仅发起 **1 次** 官方查账请求。
- **异步削峰**: 虽然派发任务在 23:00 瞬间完成，但消息消费是平滑分布在 Queue 管道中的。这确保了 D1 数据库不会因为瞬间的“重刷爆发”而产生事务冲突。
